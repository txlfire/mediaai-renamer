# MediaAI Renamer Agent 开发规范

本文是 MediaAI Renamer 项目的开发约束文档，用于指导 Codex / 大模型 / 人工开发者生成和修改代码。

项目显示名统一为 `MediaAI Renamer`，仓库名、包名、镜像名使用 `mediaai-renamer`，中文名为 `MediaAI 智能影视重命名工具`。

## 1. 项目定位

MediaAI Renamer 的核心目标是：将下载得到的混乱影视文件名整理为主流媒体刮削器容易识别的标准命名，以便 NAS、fnOS、Emby、Jellyfin、Plex、Kodi 等系统挂载和刮削。

本工具不是媒体库系统，不做播放、转码、账号体系、云同步、付费模块、第三方分享或推送通知。

后续版本可以预留 STRM 生成能力，但初版不实际生成 STRM，不修改 STRM 文件内部链接文本。

## 2. 技术栈

后端：

- Python 3.11+
- FastAPI
- SQLite
- TOML 配置
- Docker / Docker Compose

前端：

- Vue 3
- Vite
- TypeScript
- Element Plus

- Axios

部署目标：

- 普通 Linux 服务器
- NAS / fnOS 环境
- Docker Compose 服务
- Web 浏览器访问

## 3. 阶段边界

开发必须遵守里程碑边界，不提前实现大范围功能。

M0 项目骨架：

- FastAPI 后端骨架
- Vue 3 + Element Plus 前端骨架
- SQLite 初始化
- Docker Compose 基础部署
- `/api/health`

M1-M3 最小可用核心版：

- 本地或已挂载目录扫描
- 已识别目录保存
- 视频格式识别，包含常见老格式
- 文件名清洗和中文解析
- 标准命名预览
- 用户确认后安全重命名
- 页面日志查看和 TXT 导出

M4-M6 智能增强版：

- TMDB 主流程匹配
- 本地解析失败列表
- 用户确认后才调用 AI
- DeepSeek 优先接入，保留 OpenAI-compatible 配置
- 调用前估算 token
- 全量扫描 / 增量扫描
- 操作标记和扫描提示

M7-M9 NAS 与扩展版：

- Windows SMB / NAS SMB/CIFS / Linux NFS 共享识别
- 可选程序内挂载接口
- Bangumi、豆瓣代理、TVDB 兜底匹配
- 多语言识别接口
- 插件机制
- STRM 生成预留接口

## 4. 业务硬规则

1. 命名标准必须面向 TMDB、Plex、Kodi、Emby 等主流刮削器。
2. 分隔符统一使用英文句点 `.`。
3. 年份必须尽量携带，用于区分同名作品。
4. 剧集、动漫、综艺、纪录片分集统一使用 `SxxExx`。
5. 特辑、OVA、花絮统一使用 `S00Exx`。
6. 禁止文件名包含 Windows / Linux 常见非法字符。
7. 分辨率、编码、来源信息放在末尾，不干扰刮削匹配。
8. 多碟影片使用 `CD1`、`CD2` 后缀。
9. 程序先本地解析；无法解析的文件汇总到页面列表，由用户决定是否调用 AI。
10. AI 调用前必须展示大致 token 估算。
11. 重命名前必须预览，必须由用户确认。
12. 批量任务中单个文件失败不应中断整体任务，但系统级错误必须停止任务并提示。

## 5. 后端结构规范

固定目录结构：

```text
backend/app/
├── api/          # 路由控制器，只负责参数接收、调用服务、返回结果
├── core/         # 配置、数据库、异常、中间件、全局基础设施
├── schema/       # Pydantic 请求和响应模型
├── service/      # 业务服务层，放核心流程
├── utils/        # 通用工具，如文件名清洗、路径处理、日志处理
└── main.py       # 应用入口
```

后端规则：

- API 前缀统一为 `/api`。
- 控制器不得直接操作文件、数据库细节、LLM 或刮削接口。
- 文件 IO、网络请求、AI 调用必须有异常处理和日志。
- 密钥、接口地址、共享路径、模型名必须从配置读取，禁止硬编码。
- 新增依赖必须说明用途，并同步更新依赖文件和验证命令。
- 业务接口优先使用统一响应模型；`/api/health`、OpenAPI、WebSocket 可以例外。
- 配置文件不得明文提交真实密钥。

## 6. 前端结构规范

固定目录结构：

```text
frontend/src/
├── api/          # Axios 实例和业务接口封装
├── components/   # 通用组件和业务组件
├── router/       # 路由
├── stores/       # Pinia 状态
├── utils/        # 通用工具
├── views/        # 页面
├── App.vue
└── main.ts
```

前端规则：

- 后端请求统一通过 `src/api/` 中的 Axios 客户端封装。
- Axios 基础路径使用相对地址 `/api`，禁止硬编码主机和端口。
- 页面不得直接写裸 Axios 请求。
- 全局状态放入 Pinia，并按业务拆分。
- 表单必须使用 Element Plus 校验。
- 扫描、重命名等长任务进度后续优先使用 WebSocket；WebSocket 前端地址使用同源相对路径 `/ws/progress`，由部署层代理。

## 7. 命名规范

Python：

- 文件、模块、函数、变量：`snake_case`
- 类：`PascalCase`
- 常量：`UPPER_SNAKE`
- 私有方法和私有属性：单下划线前缀

TypeScript / Vue：

- Vue 页面和组件：`PascalCase.vue`
- TypeScript 文件：`camelCase.ts` 或按现有目录语义命名
- CSS 类名：`kebab-case`
- Pinia store 文件按业务命名，例如 `scanStore.ts`

## 8. 注释规范

注释使用中文。注释应解释业务意图、边界条件和风险，不重复代码表面含义。

硬性要求：

- 顶层公共注释使用中文，包括模块说明、类说明、公共函数和公共方法说明。
- 行内注释使用中文。
- 变量名、函数名、类名、文件名仍使用英文命名，保持代码国际化和生态兼容。

必须添加注释的位置：

1. 关键文件顶部：说明该文件负责的业务边界。
2. 对外公共函数 / 方法：说明用途、主要参数、返回结果和异常。
3. 核心业务流程：例如扫描、解析、匹配、预览、重命名、回滚。
4. 高风险操作：例如文件改名、路径校验、共享挂载、数据库写入、AI 调用。
5. 插件接口和抽象基类：说明扩展约束和兼容要求。

不建议添加注释的位置：

- 简单赋值。
- 一眼可见的条件判断。
- 与代码不一致或过期的解释。

Python 公共函数示例：

```python
def normalize_media_filename(raw_name: str) -> str:
    """清洗原始媒体文件名，并将分隔符统一为英文句点。

    Args:

    Returns:
        清洗后的文件名主体。

    Raises:
        ValueError: 文件名为空或只包含无效字符。
    """
```

## 9. 错误处理与日志

- 页面必须能查看任务日志。
- 日志必须支持导出 TXT。
- 日志不得记录真实密钥、完整 token、密码或敏感凭据。
- 单文件失败记录文件路径、目标名、失败原因，继续处理后续文件。
- 数据库不可写、目标目录不可访问、配置缺失、权限不足等系统级错误应终止当前任务。
- 后端使用标准 logging，不使用 `print()` 输出业务日志。
- 后端日志统一从 `app.core.logger` 获取，不直接手写 `logging.basicConfig()`。
- 日志目录、级别、文件大小和备份数量必须从配置读取。
- 项目标准日志文件：
  - `app.log`：应用启动、配置加载、普通业务流程。
  - `error.log`：ERROR 及以上错误。
  - `llm.log`：AI 模型、token 数、耗时、费用估算等统计信息，不记录 prompt 原文。
  - `batch.log`：扫描、预览、重命名等批量任务历史。
- 日志级别使用规则：
  - `DEBUG`：开发调试信息，不记录敏感内容。
  - `INFO`：关键业务节点和正常状态变化。
  - `WARNING`：可恢复异常或降级处理。
  - `ERROR`：单项功能失败，但服务仍可继续运行。
  - `CRITICAL`：配置缺失、数据库不可用等导致任务或服务无法继续的严重错误。
- Docker 部署时必须将日志目录挂载到宿主机，便于 NAS 页面查看和导出。

## 10. AI 与多源匹配

主流程：

1. 优先使用本地文件名解析。
2. 优先 TMDB 匹配，现有 `tmdb_match.py` 逻辑保持兼容。
3. TMDB 失败后按类型分流：
   - 动漫文件：Bangumi 二次匹配。
   - 国产影视 / 港台剧：豆瓣代理接口兜底。
   - 纯海外美剧：TVDB 补充分季分集信息。
4. 仍无法解析时进入“待 AI 解析”页面列表。
5. 用户确认后再调用 AI。

LLM 规则：

- 初版优先接入 DeepSeek。
- 保留 OpenAI-compatible 接口，便于后续接入 GPT、通义、智谱等模型。
- Token 价格和模型价格不得写死在代码里，应作为配置或价格表维护。
- AI 返回结果必须二次校验，不能直接用于重命名。

## 11. 插件机制

后续较大功能优先通过插件接口扩展。

预留插件类型：

- `metadata_provider`：元数据匹配源，例如 Bangumi、TVDB、豆瓣代理。
- `llm_provider`：AI 模型适配器。
- `filename_parser`：多语言文件名解析器。
- `strm_generator`：STRM 生成器，仅预留接口。
- `share_provider`：共享协议适配器，例如 SMB、NFS。

插件失败不得导致主任务崩溃，必须返回可记录、可展示的错误结果。

## 12. 文档规范

文档单独存放在 `docs/` 目录。

推荐结构：

```text
docs/
├── design/       # 项目设计文档、架构设计、数据流设计
├── development/  # 开发说明、环境搭建、编码规范、测试说明
└── api/          # API 说明，后续按模块拆分
```

开发完成后必须同步更新：

- 项目设计文档：说明架构、模块边界、关键流程、数据流和部署方式。
- 开发文档：说明环境搭建、启动方式、测试命令、代码规范、提交前检查。

## 13. Git 与提交

提交信息建议使用：

```text
[类型]: 简短中文描述
```

常用类型：

- `[feat]` 新增功能
- `[fix]` 修复问题
- `[docs]` 文档更新
- `[refactor]` 重构
- `[test]` 测试
- `[chore]` 工程配置

提交前必须运行与本次变更相关的测试和构建命令，并在说明中记录验证结果。
