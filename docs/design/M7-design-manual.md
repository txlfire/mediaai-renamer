# MediaAI Renamer M7 设计手册

版本：草案
阶段：M7 增量扫描
更新日期：2026-07-03

## 1. 阶段目标

M7 将“增量扫描”列为下一阶段主线能力，用于减少重复扫描同一媒体源时的无效文件遍历和重复预览生成。

目标：

- 第二次及后续扫描同一媒体源时，可选择全量扫描或增量扫描。
- 增量扫描只处理新增、修改、恢复可见或状态需要重新处理的文件。
- 未变化文件不重复写入扫描结果，不重复生成命名预览。
- 重命名成功后更新扫描索引和文件操作标记，避免旧路径、旧文件名造成后续扫描误判。
- 扫描结果、命名预览和待处理文件列表保持可追踪、可解释。
- 重命名页补齐 AI 批量匹配能力，并支持 TMDB 匹配失败后按用户选择自动进入 AI 二次匹配。
- 重命名预览支持在文件名信息不足时，从所属文件夹名称补充电影名或剧集名。

## 2. 范围

### 2.1 本阶段实现

- 扫描索引表。
- 文件指纹快照。
- 全量 / 增量扫描模式。
- 扫描前历史索引检测。
- 增量扫描跳过未变化文件。
- 新增、修改、删除、已重命名文件的状态标记。
- 扫描任务摘要中展示新增、修改、跳过、删除或失效数量。
- 前端扫描任务页面支持选择全量或增量扫描。
- 重命名页支持单独 TMDB 匹配、单独 AI 匹配、TMDB 未匹配时自动 AI 二次匹配。
- 批量 AI 匹配和串联匹配执行前必须显示详尽提示，说明将提交的字段范围、目标服务、敏感词保护、预计处理数量、失败回退方式和不会自动重命名。
- 命名预览解析增强：文件名缺少可用标题时，可从父级文件夹名提取标题、年份、季信息，并在预览记录中保留解析来源说明。
- 重命名预览列表增加“排除”操作，允许用户将文件从预览列表手动转入待处理文件列表。
- 最小扫描文件阈值必须继续生效：低于阈值的视频只进入待处理列表，不进入正常扫描结果和命名预览。
- 自动化测试覆盖索引写入、跳过逻辑、重命名后索引更新和回归流程。

### 2.2 本阶段不实现

- 文件内容哈希全量计算。
- 后台常驻目录监听。
- WebSocket 实时扫描进度。
- 跨媒体源去重。
- 自动删除数据库历史记录。
- 自动回滚重命名。

## 3. 核心概念

### 3.1 文件指纹

增量扫描以轻量文件指纹判断文件是否变化：

```text
media_source_id + normalized_path + file_size + modified_at + extension
```

不默认计算大文件哈希，避免 NAS / SMB / NFS 场景下产生高 IO。

### 3.2 扫描索引

扫描索引用于保存上一次已知文件状态。

建议新增表 `scan_file_index`：

| 字段 | 说明 |
| --- | --- |
| id | 主键 |
| media_source_id | 媒体源 ID |
| file_path | 当前文件完整路径 |
| normalized_path | 规范化路径，用于比较 |
| file_name | 文件名 |
| extension | 扩展名 |
| file_size | 文件大小 |
| modified_at | 文件修改时间 |
| fingerprint | 轻量指纹 |
| last_scan_job_id | 最近扫描任务 ID |
| last_seen_at | 最近一次扫描看到该文件的时间 |
| status | active / missing / renamed / ignored |
| rename_preview_id | 关联命名预览 ID，可为空 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

索引唯一性：

```text
media_source_id + normalized_path
```

### 3.3 扫描模式

`scan_jobs` 建议新增字段：

- `scan_mode`：`full` 或 `incremental`。
- `new_count`：新增文件数。
- `changed_count`：修改文件数。
- `skipped_count`：跳过未变化文件数。
- `missing_count`：历史存在但本次未发现文件数。
- `indexed_count`：本次参与索引比对文件数。

## 4. 后端流程

### 4.1 全量扫描

全量扫描流程：

1. 遍历媒体源目录。
2. 过滤非视频文件和不满足扫描配置的文件。
3. 写入或更新 `media_files`。
4. 更新 `scan_file_index`。
5. 将本次未看到但历史存在的索引标记为 `missing`。
6. 返回扫描任务摘要。

全量扫描仍然是权威刷新方式，适用于目录大量外部变更、历史索引异常或用户希望重新校验全部文件的场景。

### 4.2 增量扫描

增量扫描流程：

1. 遍历媒体源目录并生成轻量指纹。
2. 查询 `scan_file_index` 中同媒体源、同规范路径的历史记录。
3. 无历史记录：作为新增文件处理。
4. 指纹不同：作为修改文件处理。
5. 指纹相同且状态为 `active`：跳过。
6. 历史存在但本次未看到：标记为 `missing`。
7. 对新增和修改文件写入 `media_files`，后续可生成命名预览。
8. 返回新增、修改、跳过、缺失统计。

### 4.3 重命名后的索引更新

真实重命名成功后，必须同步更新扫描索引：

- 原路径索引标记为 `renamed`，记录最近任务信息。
- 新路径写入或更新为 `active`。
- 文件名、路径、大小、修改时间和指纹使用目标文件当前状态。
- 关联对应的 rename operation item，便于后续审计。

如果重命名后无法读取目标文件状态，应记录警告，不影响已完成的真实重命名结果。

## 5. 前端设计

### 5.1 扫描任务页面

媒体源选择后，如果后端检测到该媒体源存在扫描索引，页面展示扫描模式：

- 全量扫描。
- 增量扫描。

默认建议：

- 首次扫描：默认全量扫描。
- 存在索引：默认增量扫描。
- 用户最近一次扫描失败或索引异常：提示建议全量扫描。

### 5.2 扫描结果展示

扫描任务列表和详情增加统计：

- 新增。
- 修改。
- 跳过。
- 缺失。
- 已索引。

扫描结果页面默认只展示本次新增或修改后写入的结果。后续可增加“查看全部索引文件”入口，本阶段不做。

### 5.3 重命名匹配模式

M7 在重命名页增加 AI 批量匹配模式，并与 TMDB 匹配串联：

- 单独 TMDB 匹配：保持现有单条、选中批量、全部未匹配入口。
- 单独 AI 匹配：对单条、选中批量或全部未匹配条目执行 AI 结构化解析。
- TMDB 后备 AI：先执行 TMDB 匹配；仅对 TMDB 未匹配、低置信或被用户配置为需二次确认的条目执行 AI 解析。
- 串联匹配不自动执行真实重命名，不自动覆盖用户手动编辑结果；批量结果保留逐条回填入口。
- 命中敏感词或外部提交保护规则的条目不得提交到 TMDB、IMDb、AI 或其他外部服务，必须记录拦截原因。

执行前提示必须包含：

- 即将处理的文件数量。
- 将使用的外部服务：TMDB、AI 或 TMDB 后备 AI。
- 将提交的文本范围：文件名、路径、解析标题或用户选择的匹配源。
- 敏感词保护说明：命中规则时会跳过外部提交并写入拦截记录。
- 数据回填说明：批量匹配只生成候选或解析结果，默认不真实重命名。
- Token 和网络成本提示：AI 批量解析会消耗模型额度，处理数量越多成本越高。
- 失败处理方式：单条失败不阻断其他条目，最终展示成功、失败、拦截和跳过摘要。

### 5.3.1 AI Provider 抽象与模型切换

M7 需要为后续接入其他 AI 服务预留统一接口，避免批量 AI 匹配逻辑直接绑定 DeepSeek。

设计目标：

- 后端提供统一 AI Provider 抽象，批量解析、单条解析、连接测试和后续其他 AI 能力都通过同一接口调用。
- 默认保留 DeepSeek 配置，同时支持 OpenAI-compatible Provider，例如 OpenAI、硅基流动、本地兼容服务或用户自建代理。
- 设置页提供“模型切换”能力，效果类似 `cc-switch`：用户可在已保存的模型配置之间一键切换当前默认模型。
- 切换模型只改变后续 AI 调用使用的 Provider / Base URL / Model / API Key 引用，不修改历史解析结果。

后端建议接口：

```http
GET /api/settings/ai-providers
POST /api/settings/ai-providers
PUT /api/settings/ai-providers/{provider_id}
DELETE /api/settings/ai-providers/{provider_id}
POST /api/settings/ai-providers/{provider_id}/activate
POST /api/settings/ai-providers/{provider_id}/test
```

Provider 配置建议字段：

| 字段 | 说明 |
| --- | --- |
| id | Provider 主键 |
| name | 用户自定义名称，例如 `DeepSeek 默认`、`本地代理` |
| provider_type | `deepseek` / `openai_compatible` / `custom` |
| base_url | API 根地址 |
| model_name | 当前模型名 |
| api_key | 加密保存，列表接口只返回脱敏状态 |
| enabled | 是否启用 |
| is_active | 是否为当前默认模型 |
| timeout_seconds | 请求超时时间 |
| max_retries | 最大重试次数 |
| created_at / updated_at | 审计时间 |

兼容规则：

- AI 批量匹配默认使用 `is_active=true` 的 Provider。
- 执行批量 AI 或 TMDB 后备 AI 前，确认弹窗必须展示当前激活模型名称、Provider 类型、Base URL 脱敏展示和预计处理数量。
- 用户切换模型后，只影响新请求；已缓存的候选结果保留原始 `provider_id`、`provider_name`、`model_name`。
- 如果当前激活 Provider 不可用，AI 匹配入口必须禁用或要求用户先完成连接测试。
- 所有 Provider 都必须走敏感词和外部提交保护；不得绕过 M6 的拦截逻辑。
- 自定义 Provider 只支持 OpenAI Chat Completions 兼容响应；不在 M7 中实现各厂商私有协议。
- Token 用量统计按 Provider 和模型汇总，便于用户评估成本。

### 5.4 所属文件夹名称辅助解析

M7 增加“从所属文件夹名补充标题”的解析能力，用于处理文件名过短、只有集号或只有技术参数的场景。

典型场景：

- 剧集文件名为 `S01E03.mkv`、`EP03.mkv`、`03.mkv`，父级文件夹为 `权力的游戏 第一季`。
- 电影文件名为 `1080p.mkv`、`BDRip.mkv`，父级文件夹为 `盗梦空间.2010`。
- 文件名只包含清晰度、压制组、音频编码等技术信息，无法得到可靠标题。

解析优先级：

1. 文件名能解析出明确标题、年份或季集信息时，优先使用文件名。
2. 文件名只有季集信息或标题为空时，从父级文件夹名补充标题。
3. 文件名和文件夹名都包含标题但不一致时，不自动覆盖文件名标题，标记为需检查并记录原因。
4. 父级文件夹名命中敏感词或外部提交保护规则时，后续 TMDB、IMDb、AI 匹配仍必须被拦截。

后端需要在命名预览记录中保留解析来源说明，例如：

- `title_source=file_name`
- `title_source=parent_folder`
- `title_source=mixed`

前端在详情或状态提示中展示来源原因，便于用户判断是否需要手动编辑。

识别模式兼容规则：

- 新增“标题识别来源模式”配置，建议枚举为 `file_name_first`、`parent_folder_fallback`、`parent_folder_first`、`manual_only`。
- 默认使用 `parent_folder_fallback`：优先解析文件名，文件名标题不足时才使用上级文件夹补充标题。
- 剧集默认启用上级文件夹兜底识别，兼容常见目录结构：`剧名/Season 01/S01E03.mkv`、`剧名 第一季/03.mkv`。
- 电影默认只在文件名缺少有效标题时使用上级文件夹，避免电影文件名和文件夹名不一致时误覆盖。
- `manual_only` 模式不自动使用文件夹名，只在详情中展示可用候选，由用户手动确认回填。
- 全量扫描和增量扫描必须使用同一套识别模式；增量扫描跳过未变化文件时，不重新生成识别结果。
- 文件指纹变化、新增文件、用户手动重新生成预览时，按当前识别模式重新解析。
- TMDB / IMDb / AI 匹配时必须携带标题来源字段，允许用户选择使用“原文件名”“解析标题”或“上级文件夹标题”作为提交文本。
- 上级文件夹标题命中敏感词或外部提交保护规则时，即使文件名本身未命中，也不得提交到 TMDB、IMDb、AI 或其他外部服务。
- 预览记录需要保存 `title_source`、`parent_folder_title`、`recognition_mode` 和冲突原因，便于刷新、分页和批量操作后仍能追踪来源。

### 5.5 预览列表手动排除

M7 在重命名预览列表增加“排除”操作，用于用户主动将某个文件从正常预览流转到待处理文件列表。

行为规则：

- 单条预览行提供“排除”入口。
- 选中多条预览时提供批量排除入口。
- 排除前必须二次确认，说明该操作只调整任务记录，不删除真实文件。
- 排除后对应 `rename_previews` 不再出现在预览列表。
- 对应真实文件写入 `pending_files`，原因建议使用 `manual_excluded`。
- 待处理文件列表展示“手动排除”原因，用户后续可手动移动、清理或 AI 解析。
- 排除操作不执行真实重命名，不删除源文件，不影响已完成的重命名记录。

### 5.6 最小扫描文件阈值回归

M7 继续保留 M6 的扫描阈值行为：

- `scan.minimum_file_size` 以字节保存，前端按 MB 输入和展示。
- 新扫描任务中，低于阈值的视频文件不得写入 `media_files`。
- 低于阈值的视频文件进入 `pending_files`，原因使用 `size_filtered`。
- 旧扫描任务已经写入的历史小文件不自动清理；用户可通过全量重扫或后续清理能力处理。
- 扫描结果页面需要明确区分正常扫描结果和待处理文件列表，避免用户误以为阈值未生效。

## 6. API 设计

### 6.1 创建扫描任务

接口保持：

```http
POST /api/scan-jobs
```

请求体增加：

```json
{
  "media_source_id": 1,
  "scan_mode": "incremental"
}
```

兼容规则：

- 未传 `scan_mode` 时默认 `full`，保持现有调用兼容。
- 首次扫描即使传入 `incremental`，后端可降级为 `full` 并返回说明。

### 6.2 扫描模式建议

新增建议接口：

```http
GET /api/media-sources/{source_id}/scan-mode-advice
```

响应示例：

```json
{
  "media_source_id": 1,
  "has_index": true,
  "recommended_mode": "incremental",
  "reason": "检测到 235 条历史扫描索引，可使用增量扫描。"
}
```

## 7. 数据兼容

迁移要求：

- 新增 `scan_file_index` 表。
- `scan_jobs` 增加扫描模式和统计字段。
- 旧数据库迁移后历史索引为空。
- 旧前端调用不传 `scan_mode` 时仍可创建全量扫描任务。

历史数据处理：

- 不从旧 `media_files` 自动回填索引，避免误判历史路径状态。
- 用户第一次升级后执行一次全量扫描，建立索引。

## 8. 验收标准

- 首次扫描可正常全量扫描并建立索引。
- 第二次增量扫描同一目录时，未变化文件计入跳过，不重复进入扫描结果。
- 新增文件会进入本次扫描结果。
- 修改文件大小或修改时间后会进入本次扫描结果。
- 删除或移走的历史文件会标记为缺失。
- 真实重命名成功后，索引从原路径转移到目标路径。
- 前端存在历史索引时提示可使用增量扫描。
- 后端测试、前端测试、前端构建和编码检查通过。

## 9. 风险和处理

- NAS / SMB / NFS 修改时间精度不同：指纹比较时需要统一时间格式，必要时允许小范围容差。
- 远程共享目录遍历成本仍然存在：M7 只减少数据库写入和后续预览处理，不承诺避免目录遍历。
- 外部程序移动文件可能导致索引缺失：缺失状态保留，不自动删除历史。
- 文件被占用或权限不足：沿用 M5 协议校验和扫描错误记录，不中断整个任务。
