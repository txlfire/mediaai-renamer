## 2. 产品目标

MediaAI Renamer 用于将下载、转存或共享目录中的影视文件整理成主流媒体服务器可识别的标准文件名。

核心使用场景：

- Windows 本地目录整理。
- NAS / fnOS / Linux 挂载目录整理。
- Emby / Jellyfin / Plex / Kodi 入库前整理。
- 批量重命名前预览、检查和确认。
- 对本地规则无法识别的文件，后续通过 TMDB 或 AI 辅助解析。

## 4. 阶段回溯

### M1 本地目录扫描与页面框架

完成内容：

- 媒体源管理。
- 本地或已挂载目录扫描。
- 分批扫描。
- 扫描任务与扫描结果记录。
- 操作日志入口。
- 亮暗主题、侧栏和主操作台布局。

偏差和遗留：

- 只支持服务端本机可访问路径。
- 不保存 SMB / NFS 凭据。
- 增量扫描未落地。

### M2 标准命名与预览

完成内容：

- 本地文件名解析。
- 电影和剧集标准命名模板。
- 命名预览表。
- 目标文件名编辑。
- 分页、排序、长文本展示。

偏差和遗留：

- 高级命名模板配置未落地。
- 目标目录规划未落地。

### M3 安全重命名

完成内容：

- dry-run。
- 目标冲突检测。
- 批量、全部和单条重命名。
- 空文件名剔除。
- 执行批次和明细记录。

偏差和遗留：

- 不支持回滚。
- 不支持权限审批。
- 共享目录写权限校验还不完整。

### M4 TMDB 元数据与文件运维

完成内容：

- `system_settings` 热配置。
- 轻量 schema migration。
- TMDB V4 令牌优先，V3 API Key 降级。
- TMDB 匹配、候选、回填。
- 待处理文件列表。
- 最小扫描文件阈值。
- 待处理文件移除、清空和迁移。

偏差和遗留：

- 仅 TMDB 主站真实落地。
- 多站点架构是预留，不是已完成能力。
- 系统设置中非 TMDB 分类仍需 M5 落地。

### M5 NAS / SMB / NFS 共享目录支持

完成内容：

- 明确支持三类媒体源路径：`local` 本地路径、`unc` Windows UNC (SMB)、`mounted_nfs` Linux / NAS / Docker / NFS 已挂载路径。
- 在扫描和重命名前做路径可用性、读写权限和连接状态校验。
- 保存 SMB 凭据用于连接测试，凭据复用媒体源自身加密字段，必须加密保存、脱敏展示、禁止写入普通日志。
- `mounted_nfs` 不显示用户名、密码输入框，不保存密码类凭据；NFS 服务器 IP、导出目录和版本只作为该类型的可选说明与校验字段；程序只校验本地挂载点，不执行系统级挂载命令。M5 暂不支持 NFS Kerberos 域认证相关配置与密钥存储。
- 补齐系统设置中的非 TMDB 分类。

范围：

- 媒体源路径类型 `local / unc / mounted_nfs`。
- 媒体源自身 SMB 凭据加密和脱敏。
- `mounted_nfs` 已挂载路径和 NFS 可选说明字段。
- 网络协议能力边界说明。
- 共享目录连接测试。
- 目录浏览。
- 扫描前校验。
- dry-run 和执行前校验。
- Docker / NAS / NFS 部署说明。
- 文件扫描配置、解析重命名配置、系统运维配置、共享目录配置。

验收边界：

- 已完成自动化测试、构建、编码检查和基础打包验证。
- 外部 NAS / SMB / NFS 实机可达性仍需在目标部署环境中验证。

不做：

- 不自动执行系统级 SMB / NFS 挂载命令。
- M5 不实现 WebDAV、FTP、SFTP、S3 协议接入；这些协议保留为后续候选能力。
- 不绕过 M3 dry-run。

### M6 AI 智能解析

目标：

- 先建立敏感词过滤和外部提交保护网关。
- 建立外部提交拦截记录和处理列表，让用户决定修改后重新匹配、忽略匹配，或强确认后覆盖规则继续提交。
- DeepSeek / AI 作为本地解析和 TMDB 后的辅助能力。
- 只对用户确认的文件调用 AI。
- AI 输出结构化候选，由用户确认后回填命名预览。

不做：

- 不自动扫描全库调用 AI。
- 不让 AI 直接重命名真实文件。
- 不记录完整 prompt 和敏感内容。

### M7 增量扫描

完成内容：

- 新增 `scan_file_index` 扫描索引和全量 / 增量扫描模式。
- 扫描任务支持新增、修改、跳过、缺失等增量统计。
- 真实重命名后联动更新扫描索引，避免旧路径被误判。
- 命名预览支持标题识别模式、上级文件夹兜底解析和冲突提示。
- 重命名页支持 TMDB、AI、TMDB 后备 AI 三种匹配模式。
- 批量 AI 默认走当前激活 Provider，并展示明确的成功失败摘要。
- 手动排除文件可直接转入待处理列表。

偏差和遗留：

- 高级命名规则配置仍后移到 M8。
- 用户权限、回滚、审计和任务治理仍后移到 M9。
- 多站点元数据源仍未落地。

## 7. 关键设计决策

### 文件操作

所有真实文件操作必须满足：

- 用户明确确认。
- dry-run 先行。
- 后端再次校验。
- 失败可记录。
- 不覆盖已有文件。

### 外部依赖

外部服务均为增强能力：

- 文件名、路径或解析标题命中敏感词时，不提交 TMDB、IMDb、后续刮削源或 AI。
- 被敏感词或后续规则拦截的条目必须保留记录，并进入拦截列表供用户处理；覆盖规则继续提交必须强确认并记录审计信息。
- TMDB 失败时保留本地解析。
- AI 失败时保留本地解析或 TMDB 结果。
- 共享目录不可用时任务失败并记录原因，不修改数据库中已有安全数据。

### 网络协议能力边界

系统必须按协议能力区分路径类型，并明确告知用户：

- SMB / UNC：M5 支持，凭据复用媒体源自身加密字段保存，用于连接测试和共享访问校验。
- Linux / NAS / Docker / NFS 已挂载路径：M5 统一使用 `mounted_nfs` 类型，由操作系统或宿主机负责挂载，程序只访问服务端可见路径；不显示用户名、密码输入框，不保存密码类凭据；NFS 依赖 UID/GID 权限，暂不支持 Kerberos。
- WebDAV、FTP / SFTP、S3：M5 不实现，作为后续候选能力；这些协议不是普通本地文件系统，重命名、冲突、失败恢复和凭据模型需要单独设计。

### 配置

配置优先级固定为：

```text
环境变量 > 数据库热配置 > 默认值
```

敏感配置：

- 不回显明文。
- 不写普通日志。
- 只允许覆写。

## 8. 数据流总览

```text
媒体源
  |
  v
扫描任务
  |
  +--> 达标视频 --> 扫描结果 --> 命名预览 --> TMDB / IMDb / AI 候选 --> 用户确认 --> dry-run --> 执行重命名
  |
  +--> 过小/待确认文件 --> 待处理列表 --> 移除 / 清空 / 迁移 / AI 解析
```

## 9. 后续推荐顺序

建议继续按以下顺序推进：

1. M8：命名规则高级配置二期。
2. M9：权限、历史、回滚、审计和任务治理。
3. M10：多站点元数据源。
4. M11：WebDAV / FTP / SFTP / S3 等远程协议扩展评估。

理由：

- M8 直接承接现有命名模板构建器和预览链路，能够先把日常命名规则做深做稳。
- M9 适合在核心扫描、匹配、重命名流程稳定后统一补齐权限、历史、回滚和任务治理。
- M10 会引入更多外部站点和候选冲突，应该放在核心规则和治理能力之后。
- M11 涉及远程写入语义、锁、凭据和失败恢复，独立成阶段更容易控制风险。
