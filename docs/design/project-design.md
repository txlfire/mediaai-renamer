# MediaAI Renamer 系统设计文档

## 1. 项目定位

MediaAI Renamer 是面向 NAS、fnOS、Emby、Jellyfin、Plex、Kodi 等场景的影视文件扫描、解析、预览和安全重命名工具。

项目不直接承担媒体库管理职责，而是在文件进入媒体库之前，完成安全、可预览、可追踪、可回退规划的标准化整理流程。

## 2. 核心原则

- 先预览，后执行：所有真实文件重命名必须经过 dry-run 和用户确认。
- 本地优先，外部增强：本地解析和安全重命名始终可用；TMDB、AI、多站点能力都是增强能力。
- 可降级：外部接口失败、网络异常、共享目录不可用时，不应破坏已有数据和本地流程。
- 可追踪：扫描、预览、匹配、重命名、失败原因和配置变更都应可记录。
- 文件安全优先：不覆盖已有文件，不绕过冲突检测，不让 AI 直接执行真实文件操作。
- NAS 友好：部署和路径设计要兼容 Windows、Linux、Docker、NAS、fnOS 和已挂载共享目录。

## 4. 总体架构

```text
Browser
  |
  | HTTP
  v
Frontend: Vue 3 + Pinia + Element Plus
  |
  | /api
  v
Backend: FastAPI
  |
  +-- SQLite
  +-- Config / hot settings
  +-- Logging
  +-- Media source service
  +-- Scan service
  +-- Filename parser
  +-- Rename preview service
  +-- Rename operation service
  +-- Metadata provider service
  +-- AI provider service
  +-- Pending file service
  +-- Shared path protocol service
```

部署形态：

- 本地开发：Vite dev server + FastAPI。
- Docker / NAS：Nginx 静态前端 + FastAPI 后端 + SQLite 数据目录挂载。
- 共享目录：M5 后支持服务端可访问的 UNC (SMB)、Linux mount、NAS / Docker volume、NFS 本地挂载点路径。

## 5. 后端模块边界

`api/`：

- 接收 HTTP 请求。
- 做基础参数校验。
- 调用 service。
- 返回标准响应。

`service/`：

- 承载扫描、解析、匹配、预览、重命名、配置和文件运维流程。
- 不直接依赖前端展示细节。

`schema/`：

- 定义请求、响应和领域模型。
- 为接口和测试提供稳定契约。

`core/`：

- 应用配置、数据库初始化、schema migration、日志基础设施。

`utils/`：

- 文件名清洗、路径处理、视频格式判断、展示格式和通用工具。

## 6. 前端模块边界

`views/`：

- 页面级组合和业务操作入口。

`components/`：

- 通用布局、表格、弹窗、分页、工具栏和可复用业务组件。

`stores/`：

- Pinia 状态管理。
- 保存查询条件、列表数据、配置状态和用户交互状态。

`api/`：

- 统一 Axios 实例。
- 页面不得直接拼接 HTTP 请求。

`locales/`：

- 页面文案集中管理。
- 已建立中文文案配置，后续可扩展多语言。

## 7. 核心业务流程

### 7.1 扫描流程

1. 用户配置媒体源。
2. 系统检查媒体源状态和路径可访问性。
3. 用户发起扫描任务。
4. 后端按批扫描目录，识别视频文件。
5. 小于最小文件阈值的视频进入待处理文件列表。
6. 达标视频写入扫描结果。
7. 页面按任务查看扫描结果。

M5 后，共享目录扫描前必须增加路径存在、可读和连接状态校验。

### 7.2 命名预览流程

1. 用户选择媒体源和扫描任务。
2. 后端基于扫描结果执行本地文件名解析。
3. 按命名模板生成目标文件名。
4. 写入命名预览表。
5. 用户可筛选、排序、编辑目标文件名。
6. 后续 TMDB、AI 或人工候选只回填预览，不直接操作真实文件。

### 7.3 TMDB 匹配流程

1. 用户在系统设置中启用 TMDB。
2. 系统优先使用 TMDB V4 只读访问令牌。
3. V4 失败时自动降级到 V3 API Key。
4. 候选结果按标题、年份、类型、季集进行 100 分制打分。
5. 高置信结果自动回填命名预览。
6. 低置信结果进入候选弹窗，由用户确认。
7. 匹配失败时保留本地解析结果。

### 7.4 安全重命名流程

1. 用户选择命名预览记录。
2. 后端创建 dry-run 操作批次。
3. 检查空目标名、非法字符、目标冲突、批次内重复、源文件缺失。
4. 用户确认后执行真实重命名。
5. 单文件失败记录失败原因，不影响后续文件继续处理。
6. 执行结果写入批次和明细记录。

M5 后，共享目录执行前必须再次检查源文件存在、目录可写、共享未断开。

### 7.5 待处理文件流程

1. 尺寸过小或后续识别低价值文件进入待处理列表。
2. 用户可查看、移除任务记录、清空列表或批量迁移文件。
3. 移除和清空只改任务记录，不删除源文件。
4. 批量迁移会移动真实文件，必须校验目标路径和冲突。

## 8. 配置体系

系统配置统一通过 `system_settings` 管理。

优先级：

```text
环境变量 > 数据库热配置 > 默认值
```

已落地：

- TMDB 配置。
- 最小扫描文件阈值。

已落地：

- 文件扫描配置。
- 解析重命名配置。
- 系统运维配置。
- 共享目录配置。

配置要求：

- 敏感值脱敏。
- 非法值后端拒绝保存。
- 配置保存后热生效。
- 已启动任务使用任务启动时记录的配置，不被中途修改影响。

## 10. 安全边界

文件操作安全：

- 不覆盖已有文件。
- 不允许空目标文件名。
- 不允许目录穿越。
- 不绕过 dry-run。
- 不让 AI 直接修改真实文件。

共享目录安全：

- 凭据不写普通日志。
- 扫描前检查可读。
- 重命名前检查可写。
- 网络断开时失败记录必须明确。
- 按协议能力区分处理：M5 媒体源路径类型落地为 `local / unc / mounted_nfs`；SMB 使用 `unc` 类型和媒体源自身加密字段保存凭据并做连接测试；Linux / NAS / Docker / NFS 已挂载路径统一使用 `mounted_nfs` 类型，NFS 只校验本地挂载点和 UID/GID 权限，不显示或保存密码类凭据，M5 暂不支持 Kerberos；WebDAV、FTP / SFTP、S3 作为后续候选协议，不在 M5 中伪装成本地路径处理。

外部服务安全：

- API Key、token、SMB 密码必须脱敏。
- 日志不记录完整 prompt 和敏感路径。
- 文件名、路径或解析标题命中敏感词时，不得提交到 TMDB、IMDb、后续刮削源或 AI。
- 被敏感词或后续规则拦截的条目必须写入拦截记录，用户可在列表中选择修改后重新匹配、不再匹配并忽略，或在强确认后无视规则继续提交；覆盖提交必须记录原因和审计信息。
- 外部服务失败只降级，不阻断本地流程。
