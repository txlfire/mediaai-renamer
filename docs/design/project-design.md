# MediaAI Renamer 项目设计文档

## 1. 项目目标

MediaAI Renamer 用于将下载得到的混乱影视文件名整理为主流媒体刮削器可识别的标准命名，服务于 NAS、fnOS、Emby、Jellyfin、Plex、Kodi 等使用场景。

项目的核心目标不是管理媒体库，而是在文件进入媒体库之前完成安全、可预览、可回溯的标准化重命名。

## 2. 设计原则

- 先预览，后执行：所有真实重命名必须由用户确认。
- 先本地解析，后 AI：本地无法解析的文件汇总给用户，用户同意后才调用 AI。
- 可回溯：扫描、解析、重命名和失败原因都必须记录日志。
- 可部署：服务必须能以 Docker Compose 方式部署到 Linux、NAS 或 fnOS 环境。
- 可扩展：多源刮削、LLM、多语言解析、STRM 生成等大功能通过插件接口扩展。

## 3. 总体架构

```text
Browser
  │
  │ HTTP / WebSocket
  ▼
Nginx / Vite Dev Proxy
  │
  ├── Frontend: Vue 3 + Element Plus
  │
  └── Backend: FastAPI
        │
        ├── SQLite
        ├── Config TOML
        ├── Logging
        ├── Media Scanner
        ├── Filename Parser
        ├── Metadata Providers
        ├── LLM Providers
        └── Rename Executor
```

## 4. 后端模块边界

`api/`：

- 接收请求。
- 做基础参数校验。
- 调用 service。
- 返回标准响应。

`service/`：

- 承载扫描、解析、匹配、预览、重命名等业务流程。
- 不直接依赖前端展示细节。

`schema/`：

- 定义请求和响应模型。
- 为接口和测试提供稳定契约。

`core/`：

- 配置加载。
- 数据库初始化。
- 全局异常和日志基础设施。
- 日志分为应用日志、错误日志、LLM 日志和批量任务日志。

`utils/`：

- 文件名清洗。
- 路径处理。
- 视频格式判断。
- token 估算等通用逻辑。

## 5. 前端模块边界

`api/`：

- 统一 Axios 实例。
- 按业务封装后端接口。
- 页面不得直接写裸请求。

`views/`：

- 页面级组合。
- 承担展示和用户操作流程。

`components/`：

- 通用组件和业务组件。
- 扫描列表、预览表格、日志面板后续放在这里。

`stores/`：

- Pinia 状态管理。
- 保存全局状态、任务状态和配置状态。

## 6. 核心流程

扫描流程：

1. 用户选择本地目录或共享目录。
2. 用户选择全量扫描或增量扫描。
3. 后端识别视频文件和目录结构。
4. 扫描结果写入 SQLite。
5. 页面展示任务状态和日志。

解析流程：

1. 程序先清洗文件名。
2. 使用本地规则识别中文、英文、年份、季集、清晰度等信息。
3. 无法解析的文件进入待处理列表。
4. 用户确认后才调用 AI。
5. AI 返回结果必须二次校验。

匹配流程：

1. 优先 TMDB。
2. 动漫失败后走 Bangumi。
3. 国产影视和港台剧失败后走豆瓣代理。
4. 海外美剧补充 TVDB 季集信息。
5. 最终结果写入预览列表，由用户确认。

重命名流程：

1. 页面展示原文件名和目标文件名。
2. 后端执行 dry-run 冲突检测。
3. 用户确认后执行真实重命名。
4. 单文件失败记录日志，继续处理后续文件。
5. 系统级错误终止任务。

## 7. 命名规则

电影：

```text
{title}.{year}
```

剧集 / 动漫 / 综艺 / 纪录片：

```text
{title}.{year}.S{season:02d}E{episode:02d}
```

日期型节目：

```text
{title}.{YYYY-MM-DD}
```

统一要求：

- 分隔符使用英文句点。
- 年份尽量保留。
- 特辑使用 `S00`。
- 多碟影片使用 `CD1`、`CD2`。
- 非法字符必须清洗。

## 8. 部署设计

Docker Compose 包含：

- `backend`：FastAPI 服务，默认端口 `8000`。
- `frontend`：Nginx 静态服务，默认端口 `8080`。
- `data`：SQLite 和运行数据挂载目录。
- `logs`：应用日志、错误日志、LLM 日志和批量任务日志挂载目录。
- `config`：配置文件挂载目录。

Nginx 负责：

- 提供前端静态文件。
- 将 `/api/` 代理到后端。
- 后续将 `/ws/` 代理到后端 WebSocket。

## 9. 扩展点

后续插件类型：

- 元数据源插件。
- AI 模型插件。
- 文件名解析插件。
- 共享协议插件。
- STRM 生成插件。

插件接口必须保证失败可记录、可展示，不得导致主任务崩溃。
