# M9 权限、历史、回滚、审计和任务治理开发计划

## 1. 阶段目标

M9 聚焦治理能力，在不改变现有扫描、预览、匹配和重命名主流程的前提下，补齐身份、权限、审计、回滚和任务治理。

目标：

- 让高风险操作具备操作者身份和权限判断。
- 让配置变更、外部提交覆盖、真实文件操作和回滚可审计。
- 让扫描、重命名和回滚等长耗时操作有结构化过程日志，支持实时查看和事后追溯。
- 让已完成的重命名批次具备可验证的反向回滚能力。
- 让用户能集中查看扫描、重命名、回滚和批量操作状态。

## 2. 当前基线

截至 2026-07-09：

- M1-M8 已形成扫描、命名预览、TMDB / AI 回填、安全重命名、增量扫描和命名规则高级配置闭环。
- M1 页面已有当前用户和退出入口占位，但没有真实登录、会话和权限守卫。
- M3 已有重命名批次和明细记录，可作为回滚计划的数据来源。
- M6 外部提交拦截记录已有用户决策和覆盖原因字段，可在 M9 接入真实操作者。
- M7 扫描索引和任务统计已具备任务治理的数据基础。
- M8 模板版本快照已降低命名规则变化对历史预览的追踪风险。

## 3. 拆分建议

### M9-0 设计、迁移和兼容策略

- 新增用户、会话、审计事件、回滚计划和回滚明细表设计。（已完成）
- 明确开发环境和生产环境认证开关策略。
- 明确旧接口兼容策略：只对敏感写操作强制登录，健康检查和静态资源不受影响。
- 增加数据库迁移测试，旧库升级后默认无用户，允许 bootstrap 管理员。（已完成）

### M9-1 认证和权限基础

- 实现本地管理员 bootstrap。（后端已完成）
- 实现登录、退出、当前用户接口。（后端已完成）
- 使用安全密码哈希，数据库只保存哈希。（已完成）
- 实现用户权限枚举，权限直接挂在用户记录上，不引入角色模型。（已完成）
- 前端增加登录页、当前用户展示和 401/403 统一处理。（已完成）
- 在系统设置中增加用户信息维护页，支持新增、编辑、停用、权限勾选和重置临时密码。（已完成）

### M9-2 敏感接口权限守卫

- 系统设置保存、AI Provider、敏感词、命名规则保存要求 `settings:write`。
- 媒体源新增、编辑、删除要求 `source:write`。
- 扫描任务创建要求 `scan:run`。
- TMDB、IMDb、AI 外部提交要求 `metadata:submit`。
- 真实重命名执行要求 `rename:execute`。
- 前端按权限禁用对应按钮，并展示中文原因。（首批已完成）

### M9-3 审计日志

- 新增审计事件写入服务。
- 登录成功、登录失败、设置保存、媒体源变更、外部提交覆盖、真实重命名执行写入审计。（已完成）
- 回滚计划创建、dry-run 和执行审计随 M9-4 回滚能力一并接入。
- 审计详情必须脱敏 API Key、token、密码和 SMB 凭据。（已完成）
- 前端新增审计日志页面，支持筛选、分页和详情查看。（已完成）

### M9-3A 操作运行日志和实时日志窗口

目标：补齐“任务执行过程可见性”，解决扫描和重命名操作后无法看到关键过程、无法定位失败原因、无法实时观察执行进度的问题。

后端：

- 新增 `operation_logs` 表，字段包含 `task_type`、`task_id`、`level`、`stage`、`message`、`progress_current`、`progress_total`、`detail_json`、`approx_bytes`、`created_at`。
- 新增操作日志服务，提供 `record_operation_log()`、`list_operation_logs()`、`summarize_operation_logs()`，写入失败不得影响主业务流程。
- 操作日志清理复用系统“通用设置”中的日志保留天数，按 `created_at` 删除过期记录；日志清理不得删除任务记录、统计和审计事件。
- 操作日志清理必须支持大小限制：
  - 总量上限：默认 128 MB，超过后按时间优先清理最早日志。
  - 单任务上限：默认 16 MB，超过后按任务内日志顺序清理最早日志。
  - 单次清理批量：默认 1000 条，避免一次删除阻塞 SQLite。
- 日志写入应记录 `approx_bytes`，按 `message + detail_json` 的 UTF-8 字节数估算，用于快速清理判断。
- 清理时尽量保留任务最终状态、error 和 warning；详细日志被部分清理时，摘要接口必须返回中文提示。
- 摘要接口必须返回日志可用状态：`logAvailable`、`cleared`、`message`，用于前端控制“查看日志”按钮和清理提示。
- 新增查询 API：
  - `GET /api/operation-logs?task_type=scan_job&task_id=1&after_id=0&limit=200`
  - `GET /api/operation-logs/summary?task_type=scan_job&task_id=1`
  - `GET /api/operation-logs/export?task_type=scan_job&task_id=1`
- 扫描任务写入：任务创建、路径校验、扫描开始、批量进度、小文件过滤汇总、单文件异常、部分完成、失败、完成。
- 重命名 dry-run 写入：批次创建、条目数量、冲突检查、空名检查、共享协议校验、可执行/冲突汇总。
- 真实重命名执行写入：执行开始、逐项失败、成功/失败汇总、索引同步结果、最终状态。
- 写入频率必须受控：扫描普通成功项不逐文件写日志，按批次、时间间隔或状态变化写入；错误、warning 和最终状态必须保留。
- 日志详情必须复用脱敏规则，不写入 API Key、token、密码、SMB secret。

前端：

- 新增可复用 `OperationLogDrawer` 或 `OperationLogDialog` 组件。
- 触发扫描后自动打开“扫描日志”窗口，默认按不低于 1500 ms 的间隔轮询增量日志；高频写入、请求失败或页面不可见时自动降频。
- 触发 dry-run 和真实重命名后自动打开“重命名日志”窗口，展示实时进度和失败原因。
- 扫描任务列表和重命名结果/批次入口增加“查看日志”图标。
- 日志窗口支持自动刷新、手动刷新、按级别筛选、复制、导出、自动贴底。
- 日志窗口最多渲染最近 1000 条日志，超过后折叠较早日志并提示隐藏数量，避免浏览器 DOM 持续增长。
- 日志已按保留策略清理时，“查看日志”按钮置灰并展示 tooltip；如果用户已打开弹窗，则显示“该任务日志已按保留策略清理”的空态提示并停止轮询。
- 任务完成、失败或取消后停止轮询，保留日志可回看。

验收：

- 扫描运行中能看到阶段、进度和最近日志。
- 扫描完成后能从扫描任务列表回看日志。
- 重命名 dry-run 能看到冲突检查和可执行汇总日志。
- 真实重命名执行能看到成功/失败汇总和失败项原因。
- 日志接口支持 `after_id` 增量拉取，避免每次刷新全量加载。
- 日志接口限制单次返回数量，默认 100，最大不超过 500。
- 日志清理后，列表入口能置灰或弹窗明确提示已清理。
- 实时日志窗口在高频日志场景下不会持续增加 DOM 节点或维持无效轮询。
- 日志内容不包含凭据、token、API Key、密码明文。

## 4. 重点设计约束

- 不绕过 M3 dry-run 和 M5 共享路径校验。
- 不让回滚覆盖已有文件。
- 不把 API Key、token、密码、SMB 凭据写入审计明细。
- 不把 API Key、token、密码、SMB 凭据写入操作运行日志。
- 不在 M9 引入企业级多租户和外部身份协议。
- 不让只读用户触发外部提交、真实重命名、回滚和配置保存。
- 开发免登录必须由明确配置控制，不能成为生产默认行为。

## 5. 不做范围

- OAuth、LDAP、OIDC、双因素认证。
- 多租户和媒体源级数据隔离。
- 跨设备会话管理。
- 审计日志远程推送。
- 对外部程序移动、覆盖或删除后的文件做强制恢复。
- WebSocket 实时任务进度；M9-3A 采用前端轮询实现近实时日志。
