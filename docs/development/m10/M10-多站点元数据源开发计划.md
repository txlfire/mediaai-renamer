# M10 多站点元数据源开发计划

## 1. 阶段目标

M10 聚焦多站点元数据源，在现有 TMDB、IMDb 补充和 AI 回填基础上，增加可配置、可降级、可追踪的多源匹配能力。

目标：

- 统一 Metadata Provider 抽象。
- 支持 Bangumi、TVDB、豆瓣代理作为补充源。
- 建立多源匹配编排、候选合并、评分和来源展示。
- 保持 M6 敏感词保护、M7 匹配来源、M8 命名模板版本快照兼容。

## 2. 当前基线

截至 2026-07-11：

- TMDB V4/V3 已落地，V4 优先、V3 降级。
- IMDb 补充已提前落地。
- 命名预览支持 TMDB、AI 和 TMDB→AI 后备链路。
- 外部提交保护已覆盖 TMDB、IMDb 和 AI。
- 候选确认回填已能刷新命名模板版本快照。
- M10-0A 已完成后端元数据源配置表、配置接口、配置级测试和脱敏保存。
- M10-0B 已完成 Provider registry、多源匹配编排服务和重命名预览多源匹配 API 入口。
- M10-1 已完成候选去重、来源优先级、字段完整度和批量 Provider 摘要。
- M10-2 已完成 Bangumi 官方 API 真实搜索、连接测试、候选映射、请求重试和失败隔离。
- M10-3 已完成 TVDB 官方 API 真实搜索、登录、连接测试、剧集搜索和季集信息补充。
- M10-4 已完成用户自备豆瓣代理后端搜索、连接测试、候选映射和失败隔离。
- M10-5 已完成补充源前端配置入口、重命名页多源匹配入口和候选字段来源展示。

## 4. 重点设计约束

- TMDB 仍是默认主站，不因 M10 改写现有 TMDB 基础流程。
- 豆瓣代理不得写死具体第三方地址。
- 多源匹配不得绕过敏感词和外部提交保护。
- 单站点失败不能阻断本地解析、TMDB 或其他站点结果。
- 候选确认回填后必须继续走命名模板构建器。
- 多源候选必须展示来源，不能让用户误以为全部来自 TMDB。
- 当前通用凭据加密沿用历史兼容实现，后续需统一设计版本化强加密迁移，不能只迁移单一 Provider。

## 5. 不做范围

- 不做 WebDAV、FTP、SFTP、S3 远程协议。
- 不抓取未授权网页。
- 不做媒体库入库管理。
- 不做多站点账号共享市场。
- 不做企业级凭据管理和轮换策略。
