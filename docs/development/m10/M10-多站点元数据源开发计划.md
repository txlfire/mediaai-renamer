# M10 多站点元数据源开发计划

## 1. 阶段目标

M10 聚焦多站点元数据源，在现有 TMDB、IMDb 补充和 AI 回填基础上，增加可配置、可降级、可追踪的多源匹配能力。

目标：

- 统一 Metadata Provider 抽象。
- 支持 Bangumi、TVDB、豆瓣代理作为补充源。
- 建立多源匹配编排、候选合并、评分和来源展示。
- 保持 M6 敏感词保护、M7 匹配来源、M8 命名模板版本快照兼容。

## 2. 当前基线

截至 2026-07-09：

- TMDB V4/V3 已落地，V4 优先、V3 降级。
- IMDb 补充已提前落地。
- 命名预览支持 TMDB、AI 和 TMDB→AI 后备链路。
- 外部提交保护已覆盖 TMDB、IMDb 和 AI。
- 候选确认回填已能刷新命名模板版本快照。
- 多站点真实接入仍未落地。

## 3. 拆分建议

### M10-0 Provider 抽象和配置模型

- 抽象 Metadata Provider 接口。
- 新增或整理元数据源配置模型。
- 支持启用、禁用、优先级、Base URL、API Key、超时和重试。
- 保持现有 TMDB 配置兼容。
- 增加配置脱敏和连接测试模型。

### M10-1 多源匹配编排

- 新增多源匹配编排器。
- 支持 `primary_only`、`fallback`、`parallel` 三种模式。
- 站点失败隔离，不阻断其他站点。
- 复用外部提交保护，命中敏感词时不得发起任何外部请求。
- 批量多源匹配返回成功、失败、拦截和跳过摘要。

### M10-2 Bangumi 补充源

- 实现 Bangumi Provider。
- 支持动漫和番剧标题搜索。
- 输出统一候选结构。
- 在匹配原因中说明 Bangumi 来源和命中依据。

### M10-3 TVDB 补充源

- 实现 TVDB Provider。
- 支持剧集和季集信息补充。
- 输出 season、episode、episode title 等结构化字段。
- 与 TMDB 候选合并时保留字段来源。

### M10-4 豆瓣代理补充源

- 实现豆瓣代理 Provider。
- Base URL 必须由用户配置，系统不内置第三方代理地址。
- API Key 或 Token 按用户代理要求可选配置并脱敏保存。
- 未配置 Base URL 时给出明确中文提示，不发起请求。
- UI 显示用户自备代理和合规风险说明。

### M10-5 候选合并、缓存和前端展示

- 建立多源候选去重和 `merge_key`。
- 扩展候选缓存，保留 provider、provider_id、来源优先级和字段完整度。
- 候选弹窗展示来源标签、字段来源、分数和原因。
- 批量多源匹配前展示站点、数量、敏感词保护和失败隔离说明。

### M10-6 文档和验收

- 更新 README 当前阶段和文档索引。
- 输出 M10 设计手册、验收清单和用户手册。
- 更新完整用户手册，只记录已落地的多源匹配能力。
- 执行后端测试、前端测试、前端构建、编码检查和 `git diff --check`。

## 4. 重点设计约束

- TMDB 仍是默认主站，不因 M10 改写现有 TMDB 基础流程。
- 豆瓣代理不得写死具体第三方地址。
- 多源匹配不得绕过敏感词和外部提交保护。
- 单站点失败不能阻断本地解析、TMDB 或其他站点结果。
- 候选确认回填后必须继续走命名模板构建器。
- 多源候选必须展示来源，不能让用户误以为全部来自 TMDB。

## 5. 不做范围

- 不做 WebDAV、FTP、SFTP、S3 远程协议。
- 不抓取未授权网页。
- 不做媒体库入库管理。
- 不做多站点账号共享市场。
- 不做企业级凭据管理和轮换策略。
