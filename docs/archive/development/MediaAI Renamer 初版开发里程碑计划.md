# MediaAI Renamer 初版开发里程碑计划

## 总体路线

项目名使用 `MediaAI Renamer`，仓库/包/目录名使用 `mediaai-renamer`，中文名为 `MediaAI 智能影视重命名工具`。

开发顺序采用“先能安全重命名，再逐步增强刮削和扩展”的路线。

第一阶段先完成最小可用核心版：本地/已挂载目录扫描、标准命名预览、安全重命名和日志追溯。

第二阶段接入 TMDB、DeepSeek 按需解析和增量扫描，形成真正可用的智能重命名版本。

第三阶段再增强 NAS 共享、多源兜底匹配和插件机制。

## M0：项目骨架

目标：项目能启动，前后端能通信。

交付：

- FastAPI 后端骨架
- Vue 3 + Element Plus 前端骨架
- SQLite 初始化
- Docker Compose 基础部署
- `/api/health`
- 基础配置读取

验收：

- 浏览器能打开页面。
- 后端健康检查正常。
- Docker Compose 能启动基础服务。

## M1：本地/已挂载目录扫描

目标：能扫描媒体目录并保存结果。

交付：

- 媒体源配置
- 已识别目录保存
- 视频格式识别
- 全量扫描
- 扫描任务状态
- 扫描日志页面

验收：

- 选择一个本地或已挂载目录，可以扫描出视频文件列表。
- 扫描结果能写入 SQLite。
- 页面能查看扫描任务状态。

## M2：标准命名与预览

目标：不调用外部 API，也能做基础规范化预览。

交付：

- 文件名清洗
- 中文/英文基础解析
- 标准命名模板
- 英文分隔符规范
- 非法字符处理
- 电影/剧集命名预览
- 用户手动编辑预览项

验收：

- 混乱文件名能生成可编辑的新文件名。
- 预览阶段不修改任何真实文件。
- 默认命名符合主流刮削标准。

## M3：安全重命名

目标：完成核心闭环。

交付：

- dry-run 检查
- 同名冲突检测
- 路径白名单
- STRM 只改名保护
- 批量重命名
- 重命名日志
- TXT/CSV/JSON 导出

验收：

- 用户确认后可安全重命名。
- 失败项不中断整个任务。
- 日志可追溯原路径、目标路径、执行结果和错误原因。

## M4：TMDB 主刮削

目标：主流程接入 TMDB，提高命名准确率。

交付：

- TMDB 搜索
- 电影/剧集匹配
- 匹配置信度
- 候选结果
- TMDB 失败状态
- 标准名回填预览

验收：

- 常见电影和剧集能通过 TMDB 匹配。
- TMDB 匹配结果能生成标准命名。
- 低置信度结果会提示用户确认。

## M5：DeepSeek 按需解析

目标：只在用户确认后使用 AI。

交付：

- 本地解析失败列表
- AI 解析确认页
- token 预估
- DeepSeek 接入
- OpenAI-compatible 配置
- AI 结果校验

验收：

- 扫描不会自动调用 AI。
- 用户确认后才解析失败项。
- 调用前展示 token 预估。

## M6：增量扫描

目标：提升重复扫描效率。

交付：

- 扫描索引
- 操作标记
- 增量扫描
- 全量/增量提示
- 跳过未变化文件

验收：

- 第二次扫描能提示用户选择全量或增量。
- 增量扫描能跳过未变化文件。
- 重命名成功后会更新扫描索引和操作标记。

## M7：共享源增强

目标：支持 NAS 场景。

交付：

- Windows SMB 共享识别
- NAS SMB/CIFS 共享识别
- Linux NFS 共享识别
- 连通性测试
- 已挂载共享路径支持
- 可选程序内挂载接口

验收：

- NAS/fnOS 环境下能访问共享目录并扫描。
- 页面能展示共享类型和连通性状态。
- 凭据不明文返回前端，不写入日志。

## M8：多源兜底匹配

目标：补 TMDB 的不足。

交付：

- 多语言识别接口
- 媒体类型分流
- Bangumi 动漫匹配
- 豆瓣代理兜底
- TVDB 美剧分集补充
- 匹配源标识展示

验收：

- TMDB 失败后能按类型走正确兜底源。
- 预览页能展示最终匹配源和兜底原因。
- 豆瓣代理地址可配置，不写死。

## M9：插件基础机制

目标：为后续大功能扩展留接口。

交付：

- 插件接口协议
- 插件注册表
- 插件配置命名空间
- metadata_provider 插件化
- strm_generator 预留接口

验收：

- Bangumi/TVDB 等适配器能按插件接口注册和调用。
- 插件失败不会导致主任务崩溃。
- STRM 生成能力有预留接口但初版不实际生成。

## 建议排期

单人开发：

- M0-M3：2-3 周
- M4-M6：2-3 周
- M7-M9：3-4 周
- 总计：7-10 周

两人开发：

- M0-M3：1.5-2 周
- M4-M6：1.5-2 周
- M7-M9：2-3 周
- 总计：5-7 周

## 阶段划分

第一阶段：M0-M3

产出最小可用核心版，支持目录扫描、标准命名预览、安全重命名和日志导出。

第二阶段：M4-M6

接入 TMDB、DeepSeek 按需解析和增量扫描，形成实用版。

第三阶段：M7-M9

增强 NAS 共享、多源兜底匹配和插件机制，形成可扩展版本。
